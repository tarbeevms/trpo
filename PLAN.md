# TaskFlow Plan

## Тема проекта

`TaskFlow` - простое веб-приложение на Go для управления личными задачами и проектами.

Приложение должно строго соответствовать учебному заданию и не содержать лишнего функционала.  
Интерфейс: веб-приложение с очень простым HTML, чтобы можно было вручную проверить основные сценарии.  
Хранение данных: БД PostgreSQL в Docker.

## Цель проекта

Разработать и реализовать объектно-ориентированное приложение на Go с использованием:

- принципов ООП;
- не менее 8 классов/структур предметной области;
- связей между объектами;
- простого и множественного наследования в рамках возможностей Go;
- 2 сочетающихся паттернов проектирования;
- логирования;
- загрузки и сохранения данных в БД;
- unit-тестирования;
- документации по разработке и реализации.

## Формат итогового приложения

- backend на Go;
- PostgreSQL в Docker;
- простой HTML-интерфейс без лишнего оформления;
- структура проекта по папкам:
  - `build`
  - `cmd/taskflow`
  - `frontend`
  - `internal/app`
  - `internal/models`
  - `internal/service`
  - `internal/repository`
  - `pkg/logger`
  - `readme.md`

В папке `build` должен находиться Docker-файл для поднятия PostgreSQL: `build/Dockerfile`.  
Файл `docker-compose.yml` должен использовать `build/Dockerfile` для сервиса PostgreSQL.
Точка входа приложения должна находиться в `cmd/taskflow/main.go`.

Все файлы в `internal/app`, `internal/service`, `internal/repository` должны называться по соответствующему сервису или предметной области:

- `task.go`
- `project.go`
- `user.go`
- `report.go`

В папке `internal/models` будут находиться доменные структуры, value objects и структуры для демонстрации ООП. В `models` допускаются файлы по моделям, например:

- `common.go`
- `user.go`
- `project.go`
- `task.go`
- `report.go`

Назначение основных папок:

- `cmd/taskflow` - точка входа приложения;
- `internal/app` - только HTTP-обработчики;
- `internal/models` - модели предметной области;
- `internal/service` - бизнес-логика, фасад и стратегии;
- `internal/repository` - методы работы с PostgreSQL;
- `pkg/auth` - JWT-авторизация;
- `pkg/logger` - общий логгер.

## Предметная область

Приложение позволяет:

- регистрировать пользователей;
- выполнять вход через JWT;
- выполнять выход;
- открывать персональную страницу пользователя `/me`;
- создавать проекты;
- просматривать список проектов с названием и описанием;
- создавать задачи;
- менять статус задачи;
- просматривать список задач;
- экспортировать задачи текущего пользователя в форматах HTML, JSON, XML.

## Входные и выходные данные

Веб-интерфейс будет отправлять данные через HTML-формы. HTTP-обработчики будут принимать данные формы, передавать их в сервисы, а результат отображать на HTML-странице.

Основные входные данные:

- регистрация: `login`, `password`;
- вход: `login`, `password`;
- проект: `name`, `description`;
- задача: `project_id`, `title`, `description`, `priority`, `deadline`, `tags`;
- изменение статуса: `task_id`, `status`;
- отчёт/экспорт: `format`.

`format` может быть:

- `html` - отобразить на странице;
- `json` - отобразить задачи в JSON;
- `xml` - отобразить задачи в XML.

Основные выходные данные:

- персональная страница текущего пользователя;
- список задач текущего пользователя;
- список проектов текущего пользователя (название + описание);
- сообщение об ошибке валидации;
- экспорт задач текущего пользователя в выбранном формате.

## Ограничения на данные

`User`:

- `login` - обязательный, уникальный, от 3 до 50 символов, без пробелов;
- `password` - от 6 до 72 символов;
- `password_hash` - хранится в БД вместо открытого пароля.

`Project`:

- `name` - от 3 до 80 символов;
- `description` - до 500 символов;
- `owner_id` - обязательный, должен ссылаться на существующего пользователя.

`Task`:

- `title` - от 3 до 100 символов;
- `description` - до 1000 символов;
- `status` - только `new`, `in_progress`, `done`, `cancelled`;
- `priority` - только `low`, `medium`, `high`, `critical`;
- `deadline` - не раньше текущей даты;
- `project_id` - обязательный, должен ссылаться на существующий проект;
- `assignee_id` - обязательный, должен ссылаться на существующего пользователя.

`Tag`:

- `name` - от 2 до 30 символов;
- у одной задачи может быть максимум 10 тегов.

`TaskHistory`:

- хранит `task_id`;
- хранит старый статус;
- хранит новый статус;
- хранит дату изменения.

`Report`:

- содержит тип отчета и элементы отчета в доменной модели;
- в текущем пользовательском сценарии отчет реализован как экспорт задач через `ReportExporter`.

## Как соблюдаем ООП

### 1. Инкапсуляция

Инкапсуляция означает сокрытие деталей реализации и предоставление ограниченного интерфейса для работы. В Go это достигается через:

**а) Неэкспортируемые поля (private)**
- В сервисах поля начинаются со строчной буквы: `tasks`, `projects`, `users`, `logger`, `exporters`
- Внешний код не может напрямую изменить эти поля
- Пример: [`TaskFacade`](internal/service/task.go:18-24) скрывает `tasks`, `projects`, `users`, `logger`

**б) Методы для изменения состояния**
- Изменение статуса задачи происходит через метод [`Task.ChangeStatus()`](internal/models/task.go:93-106)
- При смене статуса автоматически создаётся запись в истории
- Валидация данных происходит в методах `Validate()` моделей

**в) Конструкторы**
- Сервисы создаются через конструкторы `New...`: [`NewTaskFacade()`](internal/service/task.go), [`NewReportService()`](internal/service/report.go)
- Конструктор сразу инициализирует все зависимости в корректном состоянии

**г) Вспомогательные функции**
- Выбор экспортёра скрыт в [`ExporterRegistry.Get()`](internal/service/report_exporter.go)

### 2. Абстракция

Абстракция означает работу через общие интерфейсы без знания конкретной реализации:

- [`TaskStore`](internal/service/task.go:11-16) - интерфейс для работы с задачами
- [`ProjectStore`](internal/service/project.go:10-15) - интерфейс для работы с проектами
- [`UserStore`](internal/service/user.go:20-27) - интерфейс для работы с пользователями
- [`ReportExporter`](internal/service/report_exporter.go:13-18) - интерфейс для экспорта отчётов

### 3. Полиморфизм

Полиморфизм позволяет одному и тому же коду работать с разными реализациями:

**Экспортёры отчётов:**
- [`JSONExporter`](internal/service/report_exporter.go:21-33) - экспорт в JSON
- [`XMLExporter`](internal/service/report_exporter.go:36-55) - экспорт в XML
- [`HTMLExporter`](internal/service/report_exporter.go:58-77) - экспорт в HTML
- Все реализуют интерфейс [`ReportExporter`](internal/service/report_exporter.go:13-18)
- [`ExporterRegistry`](internal/service/report_exporter.go:81-83) хранит все реализации и выбирает нужную по формату
- [`ReportService`](internal/service/report.go) получает задачи через `TaskStore`, выбирает экспортёр и вызывает `ExportTasks()` через общий интерфейс

### 4. Наследование

В Go нет классического наследования, поэтому используем embedding (встраивание):

**Простое наследование:**
- `Task`, `Project`, `User` встраивают `BaseEntity` для полей `ID`, `CreatedAt`, `UpdatedAt`

**Множественное наследование:**
- `User`, `Project` и `Task` встраивают несколько структур, например `BaseEntity`, `AuditInfo`, `SoftDelete`
- Это позволяет одной структуре получить поля и методы от нескольких встроенных структур

### 5. Связи между объектами

- **Ассоциация:** `Task` связана с `Project` через `ProjectID`
- **Агрегация:** `Project` содержит массив задач (в коде - через внешний ключ)
- **Композиция:** `TaskHistory` принадлежит `Task`
- **Зависимость:** сервисы зависят от репозиториев и логгера

## Связи между объектами

- ассоциация:
  - `Task` связана с `User` как исполнитель;
- агрегация:
  - `Project` содержит задачи;
- композиция:
  - `TaskHistory` принадлежит задаче;
- зависимость:
  - `service` зависит от `repository` и логгера.

## Паттерны проектирования

Будем использовать 2 паттерна, которые сочетаются друг с другом:

1. `Facade` (Фасад)
2. `Strategy` (Стратегия) - используется для экспорта отчётов в разные форматы

### 1. Facade (Фасад)

**Зачем нужен:** Операции над задачей затрагивают несколько частей системы: валидацию, репозиторий, логирование, историю изменений. Фасад скрывает эту координацию за единым сервисным интерфейсом.

**Реализация:** [`TaskFacade`](internal/service/task.go:18-24) в `internal/service/task.go`

- Метод `Create()` выполняет: валидацию → проверку проекта → проверку исполнителя → сохранение → логирование
- Метод `ChangeStatus()` выполняет: поиск задачи → смену статуса → создание истории → обновление в БД → логирование

### 2. Strategy (Стратегия) - Экспорт отчётов

**Зачем нужен:** Отчёт можно экспортировать в разных форматах. Каждый формат - отдельная стратегия.

**Реализация:** Интерфейс [`ReportExporter`](internal/service/report_exporter.go:13-18) и структуры:
- [`JSONExporter`](internal/service/report_exporter.go:21-33) - экспорт в JSON
- [`XMLExporter`](internal/service/report_exporter.go:36-55) - экспорт в XML
- [`HTMLExporter`](internal/service/report_exporter.go:58-77) - экспорт в HTML

[`ExporterRegistry`](internal/service/report_exporter.go:81-83) хранит все реализации и выбирает нужную по формату.

### Почему паттерны сочетаются

- `Facade` координирует пользовательские сценарии (создание задачи, смена статуса)
- `Strategy` (экспорт) определяет формат вывода отчёта

Они решают разные задачи: фасад упрощает работу с подсистемой задач, а стратегия делает заменяемым алгоритм экспорта.

### Этап 8. КМ-5: Unit-тестирование

Unit-тестирование нужно провести для отдельных модулей исходного кода. Тесты должны проверять корректность нетривиальных функций и методов.

По требованию задания unit-тесты пишет разработчик до написания основного кода. Поэтому порядок работы для реализации должен быть таким:

1. Сначала написать тесты для нетривиальной бизнес-логики.
2. Убедиться, что тесты сначала не проходят, потому что код еще не реализован полностью.
3. Реализовать код.
4. Запустить тесты повторно и добиться успешного прохождения.
5. Зафиксировать в документации, какие тесты были написаны и что они проверяют.

Нетривиальные функции и методы для unit-тестирования:

1. Валидация `User`.
2. Валидация `Project`.
3. Валидация `Task`.
4. Валидация `Tag`.
5. Изменение статуса задачи.
6. Проверка дедлайна задачи.
7. Проверка количества тегов.
8. Экспорт задач через `ReportExporter`.
9. Создание задачи через `TaskFacade`.
10. Регистрация пользователя.
11. Вход пользователя.
12. Генерация и проверка JWT.

Позитивные тесты должны проверять корректную работу при ожидаемых данных:

- регистрация пользователя с корректными `login`, `password`;
- вход пользователя с корректными `login`, `password`;
- генерация и проверка JWT;
- создание проекта с корректными `name`, `description`, `owner_id`;
- создание задачи с корректными полями;
- смена статуса задачи на допустимый статус;
- экспорт задач в HTML;
- экспорт задач в JSON;
- экспорт задач в XML;
- проверку, что экспорт использует `AssigneeID` текущего пользователя.

Негативные тесты должны проверять обработку некорректных данных. Система не должна падать, а должна возвращать ошибку:

- пустой `title`;
- неверный `status`;
- неверный `priority`;
- дедлайн в прошлом;
- несуществующий исполнитель;
- несуществующий проект;
- некорректный `login`;
- слишком короткий пароль;
- неверный пароль при входе;
- JWT с неверной подписью;
- больше 10 тегов;
- слишком длинное описание задачи.

Граничные тесты должны проверять значения на границах допустимых диапазонов:

- `User.login` ровно 3 символа;
- `User.login` ровно 50 символов;
- `Project.name` ровно 3 символа;
- `Project.name` ровно 80 символов;
- `Task.title` ровно 3 символа;
- `Task.title` ровно 100 символов;
- `Task.description` ровно 1000 символов;
- `Tag.name` ровно 2 символа;
- `Tag.name` ровно 30 символов;
- ровно 10 тегов у задачи;
- дедлайн сегодня.

Тесты паттерна `Strategy`:

- выбор экспортёра по формату;
- экспорт задач в HTML;
- экспорт задач в JSON;
- экспорт задач в XML;
- формат по умолчанию для неизвестного значения.

Тесты паттерна `Facade`:

- успешное создание задачи через `TaskFacade`;
- отказ при ошибке валидации;
- запись истории при смене статуса;
- логирование ошибки при неуспешной операции.
